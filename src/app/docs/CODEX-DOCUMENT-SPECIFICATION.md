# CODEX DOCUMENT SPECIFICATION

**Project:** Heroes or Henchmens

**Codename:** Codex Vaticinarum

**Version:** 1.0.0

**Status:** FOUNDATION FROZEN

---

# 1. Purpose

The Codex Vaticinarum defines the structure, hierarchy and editorial language of every official document produced by The Order.

It is not a UI specification.

It is not an Angular specification.

It defines how an official archive record exists.

Every Codex document must follow this specification.

---

# 2. Document Philosophy

Every record follows five immutable principles.

## Institutional

The document belongs to The Order.

It must look official.

Never playful.

Never decorative.

---

## Information First

Information is always more important than visual elements.

Every decorative asset exists only to reinforce the archive.

---

## Hierarchy

Every section has a fixed importance.

Readers must immediately understand where information begins and ends.

---

## Consistency

Every archive follows the same structure.

Different contents.

Same document language.

---

## Permanence

Documents must feel timeless.

A Codex page should appear equally believable whether viewed today or centuries from now.

---

# 3. Document Identity

A Codex Record is an institutional archive.

It is composed of:

• Identity

• Representation

• Information

• Authentication

No document exists without these four pillars.

---

---

# 6. Header

The Header identifies the archive.

It never explains it.

The Header contains:

• Registry Code

• Classification

• Security Level

• Title

• Subtitle

The Header always occupies the upper region of the document.

---

# 7. Glyph

The Glyph represents the archived object.

It never represents the document itself.

Rules:

One primary Glyph per record.

Always centered.

Never decorative.

Never larger than its reserved area.

Never overlaps textual content.

---

# 8. Summary

The Summary answers one question:

"What is this?"

It is intentionally concise.

Maximum:

Two paragraphs.

---

# 9. Body

The Body contains the archive itself.

Possible sections:

Description

Origin

Characteristics

Properties

Known Behaviour

Warnings

Observations

Relationships

Not every document contains every section.

---

# 10. Timeline

Timeline records historical events.

Examples:

Discovery

Recovery

Classification

Containment

Relocation

Incident

Revision

Timeline entries are always chronological.

---

# 11. Tags

Tags are metadata.

They never replace classifications.

They improve indexing.

Examples:

Ancient

Stable

Organic

Mechanical

Unknown Origin

Energy

Relic

---

# 12. Footer

The Footer authenticates the archive.

Possible information:

Registry Identifier

Archive Branch

Revision

Verification

Official Signature

Footer never contains decorative elements.

---

# 13. Classification System

Every document belongs to exactly one classification.

Current classifications:

Artifact

Entity

Location

Archive

Protocol

Mission

Order

Phenomenon

Technology

Unknown

---

# 14. Security Levels

Every archive possesses one security level.

Hierarchy:

PUBLIC

↓

RESTRICTED

↓

CLASSIFIED

↓

ORDER ONLY

↓

OMEGA

Security level never changes document anatomy.

---

# 15. Information Hierarchy

Priority:

Document

↓

Information

↓

Glyph

↓

Frame

↓

Decorative Assets

Decoration must never compete with information.

---

# 16. Frame Philosophy

The Frame belongs to The Order.

Not to the Artifact.

Every document shares the same frame language.

Frames identify the institution.

Never the archived object.

---

# 17. Glyph Philosophy

Glyphs are symbolic abstractions.

They are not illustrations.

They are not icons.

They represent the conceptual identity of the archived object.

Glyph construction follows the Order Design System.

---

# 18. Composition Rules

Every section has one responsibility.

Header identifies.

Glyph represents.

Summary introduces.

Body documents.

Timeline records.

Tags classify.

Footer authenticates.

Responsibilities never overlap.

---

# 19. Validation Rules

A document is considered valid only if it contains:

✓ Header

✓ Glyph

✓ Summary

✓ Body

✓ Footer

Timeline and Tags are optional.

---

# 20. Future Extensions

Future document modules may include:

Cross References

Related Records

Personnel

Recovered Objects

Mission Reports

Expedition Logs

Incident Files

Attachments

Extensions never modify the core anatomy.

---

# 21. Golden Rule

Before introducing any new visual element, ask:

"Would The Order include this in an official archive?"

If the answer is "No",

it does not belong in the Codex.
